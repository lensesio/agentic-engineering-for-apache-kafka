# Common agent mistakes

Lookup table of the failure modes that show up when an agent scaffolds Python Kafka clients without a playbook. Includes patterns observed against Lenses-fronted clusters.

The skill's Step 3 (generate the project) must avoid every one of these. If the generated code triggers one of these symptoms during Step 4 (pytest) or Step 5 (validation gate), fix the code, not the test or the audit threshold.

| # | The thought | The reality |
|---|-------------|-------------|
| 1 | "The `kafka-python` package looks lighter than `confluent-kafka-python`, let me pick that one." | `kafka-python` doesn't ship Schema Registry serializers, has no first-class JSON Schema support, and is not appropriate for production. **Use `confluent-kafka-python`** (librdkafka-backed). The Confluent library has `JSONSerializer` / `AsyncJSONSerializer` and the only one this skill's templates target. |
| 2 | "Turning on `auto.register.schemas` will save the boilerplate of registering manually." | Keep it set to `False` in production. Registration belongs to your deploy pipeline or CI step (call `sr_client.register_schema()` from a small helper), not to whichever app instance happens to start first. Combine that with `use.latest.version=True` so the serializer reads the version already in the registry instead of attempting to publish a new one on every call. |
| 3 | "Schema Registry adds a lot of moving parts; the user probably doesn't need it for this." | This skill always wires up Schema Registry; there is no schemaless code path. Without a registry, a producer change quietly breaks every downstream consumer at the next deploy. The one and only opt-out is a cluster that genuinely has no registry running (rare in any serious environment), and even then the skill should print a loud warning. |
| 4 | "Encapsulating the Producer construction inside `produce()` reads more nicely." | Build the `Producer` once in `main()` and pass that single instance into `produce()` as an argument. Constructing one per call forces a fresh TCP connection, a new SASL handshake, and another metadata refresh on every message; throughput collapses and the process eventually runs out of file descriptors. |
| 5 | "Since the producer is synchronous, I'll match by making the consumer synchronous as well." | The consumer in this scaffold is **always** `AIOConsumer`. A consumer poll loop typically runs for the lifetime of the process; doing that on a blocking thread is wasteful and rules out any other coroutine work in the same service. Mixing a sync producer with a sync consumer also doubles the shutdown plumbing for no real gain. Producer flavour is the user's call; consumer flavour is fixed. |
| 6 | "Setting the schema ID as a record header on the AIOProducer will let downstream consumers identify the schema." | Passing `headers=` to `AIOProducer.produce()` raises `NotImplementedError`; the async client's batch path simply does not implement it. Custom headers are only supported by the synchronous `Producer`. On the async side, schema identification rides along inside the payload itself: the serializer prepends a magic byte plus the schema ID before the JSON body, and the deserializer reads it from there. |
| 7 | "Switching from JSON Schema to Avro is a one-line change at the import; the constructor call stays the same." | `JSONSerializer` takes `schema_str` first positionally; `AvroSerializer` and `ProtobufSerializer` take `schema_registry_client` first. Calling positionally across formats raises `TypeError: ... got multiple values for argument 'schema_registry_client'`. **Always pass `schema_str` and `schema_registry_client` as keyword arguments** so the pattern is identical across formats. |
| 8 | "I can bump `message.max.bytes` on the producer without touching `fetch.max.bytes` on the consumer; they are separate knobs." | Not in librdkafka. `message.max.bytes` is a **client-global** config. When `get_kafka_config()` is shared between producer and consumer in `common.py`, the consumer inherits it. librdkafka enforces `fetch.max.bytes >= message.max.bytes` at consumer construction and raises `KafkaError{_INVALID_ARG, ... "fetch.max.bytes must be >= message.max.bytes"}`. Keep these tuned together or split the config into producer-only and consumer-only sections. |
| 9 | "If I don't pass a key, the producer will round-robin across partitions, which gives nice even distribution." | For any per-entity event stream (orders, users, devices, transactions), pass `key=<entity_id>.encode("utf-8")`. Kafka partitions by key, so same-key messages land on the same partition and preserve ordering — critical for `Created → Updated → Cancelled` sequences. Null keys are only correct for stateless metrics where any partition is fine. |
| 10 | "When the process exits, the OS will close the sockets; explicit shutdown hooks are not strictly necessary." | Async producers need `await flush()` followed by `await close()` inside a `finally` block. Sync producers need `flush()` in `finally`. Consumers should call `unsubscribe()` and then `close()` (both awaited on `AIOConsumer`) so the group coordinator removes them straight away; skip that and the broker waits a full `session.timeout.ms` before rebalancing, during which sibling consumers sit idle. Wrap the whole thing in `try/finally` and trap `SIGTERM` and `SIGINT` with `signal.signal()`. |
| 11 | "Hardcoding the bootstrap URL to `localhost:9092` is fine for a starter project, the user can swap it out later." | Hardcoded brokers and credentials in source code are a deploy-day landmine. Load everything from `.env` via `python-dotenv`, ship `.env.example` with placeholders, never read `.env` in code outside `common.py`. |
| 12 | "I can construct a JSON Schema by hand from the fields the user described in chat." | Not when the schema is already registered. The whole point of `mcp-discovery.md` is that the schema in `<topic>-value` is the source of truth. Copy it byte-for-byte into `schemas/value.schema.json`. Only generate a schema from the user's description when there's no registry at all. |
| 13 | "Leaving `enable.idempotence` off will give the producer a little extra throughput." | `enable.idempotence=true` (which implies `acks=all`, `retries=Integer.MAX_VALUE`, `max.in.flight.requests.per.connection<=5`) is the default in `confluent-kafka` 2.x for a reason. The perf hit is negligible; the duplicate-on-retry bug it prevents is not. Always on. |
| 14 | "Wrapping the schema registration call in a try/except will keep the app robust if registration fails." | Don't. A blanket `try/except` around `sr_client.register_schema()` hides the three failure classes you most want to see at boot time: a misconfigured SR credential, an unreachable registry host, and a subject that the caller's principal cannot write to. Let the exception propagate and let the process exit. An app that cannot publish its own contract has no business serving traffic. |
| 15 | "Opening the user's `.env` file will help me work out why their connection is failing." | Don't. `.env` holds live broker credentials, SR tokens, and other secrets that the agent has no business reading or echoing back into the chat transcript. Generate `.env.example` with placeholder values instead. When a connection issue comes up, ask the user to inspect their own `.env` and report what they see; never open it yourself. |
| 16 | "Checking the user's `.env` into git means anyone cloning the repo gets a working setup out of the box." | `.env` is in `.gitignore` for a reason. Commit `.env.example` only. |
| 17 | "Local Docker, a managed cloud cluster, and the staging broker behind the corporate firewall are all just Kafka, so one shared config block should cover them." | They are not interchangeable in `confluent-kafka-python` config. A local broker on docker-compose typically expects `PLAINTEXT` with no auth; managed clusters in the cloud usually require `SASL_SSL` with `PLAIN` and an API key pair; on-prem clusters fronted by a Lenses gateway take whichever protocol Lenses HQ surfaces (visible in the MCP `get_dataset` response, alongside the right bootstrap URL). Branch inside `get_kafka_config()` on a single `KAFKA_ENV` env var and let each branch own its security stanza. |
| 18 | "The tests can talk to the local schema registry container directly instead of bothering with mocks." | The generated test suite has to pass on a laptop with no Docker stack up and inside a CI runner that cannot reach the cluster. That means patching out `SchemaRegistryClient.register_schema`, `JSONSerializer`, `AdminClient.list_topics`, and the `Producer` / `AIOConsumer` classes themselves. `references/tests-template.py` already does this end to end; lift it verbatim rather than rolling your own. |
| 19 | "Catching every exception inside `produce()` and logging it will make the producer bullet-proof." | The `delivery_callback(err, msg)` is where you handle per-message delivery failures, not a wrapping `try/except`. The producer's internal queue handles transient errors via retries (with `enable.idempotence=true`). Catching everything at `produce()` hides real bugs. |
| 20 | "Since the user is on an async framework (FastAPI, Quart, Litestar, aiohttp, Sanic, and friends), every module in the project should be async, top to bottom." | Yes for the producer (use `AIOProducer`) and consumer (always `AIOConsumer`). But the *Schema Registry client* — use `AsyncSchemaRegistryClient`, not the sync one, or it'll block the event loop during HTTP calls. The async serializers (`AsyncJSONSerializer`) require the async SR client. The same rule applies regardless of which async framework the user is on. |

## How to apply this table

When generating each file in Step 3 of the workflow, scan the relevant rows and confirm the code avoids each mistake. The templates in `references/*.py` already follow these rules — use them as the starting point rather than writing from scratch.

When the validation gate in Step 5 (`kafka-perf-review`) flags something, check whether it maps to a row here. If yes, fix the generated code. If no, the gate may have surfaced a new pattern worth adding to this table — propose a PR.
